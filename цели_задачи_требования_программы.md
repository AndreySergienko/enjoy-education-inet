Требования к стратегии обучения разработчика

Документ описывает требования к формированию стратегии обучения разработчика

Сергея Герасина с целью повышения уровня с Middle до уверенного Middle+ в течение трёх месяцев.

## 1. Введение

Сергей Герасин в настоящий момент находится на уровне Middle. Он способен выполнять типовые задачи, но испытывает сложности в следующих областях:

• архитектурное проектирование - нет знаний о современных подходах проектирования, отсутсвтие насмотренности и опыта в работе с сложными и эффективным архитектурными подходами внутри medium/highload проектов.
• отладка сложных проблем - на текущий момент практически каждая задача вызывает трудности в понимании. Стремится сделать максимально быстро без понимания первопричин проблем. Пример, ifyoucare решение с позиционированием кнопки публиковать. Был готов захардкодить, когда решалось всё через админку.
• принятие технических решений - как правило низкокачественные и быстрые решения
• коммуникация и soft skills - не поддерживает задачи в актуальном статусе, некорректно сообщает о проблемах в задачах.

## 2. Цели и задачи обучения

Цель программы — сформировать устойчивые навыки самостоятельной разработки, архитектурного мышления и инженерной ответственности.

За 3 месяца необходимо довести разработчика до уровня уверенного Middle+.
Ключевые признаки достижения цели:

• способность самостоятельно анализировать задачи
• умение проектировать архитектуру решения
• умение выявлять проблемы и предлагать решения(самостоятельность разработчика)
• высокая культура разработки(качество кода)
• корректная коммуникация внутри команды
• развитие коммуникационных навыков
• освоение современных инструментов разработки

## 3. Основные этапы формирования стратегии

Любая стратегия обучения состоит из следующих этапов:

1. Формирование целей и задач обучения
2. Формирование инструментов обучения
3. Формирование методик обучения
4. Формирование источников материалов Каждый этап должен быть формализован и документирован.

## 5. Методики обучения

Методика обучения должна быть основана на принципах интенсивного практического обучения.
Основные элементы методики:

• обучение через реальные задачи проекта medium/highload загрузки
• промежуточные проверки знаний в формате собеседования раз в неделю.
• обсуждение архитектурных решений

Каждая тема обучения должна сопровождаться практическим заданием.
Будь редактирование в рамках проекта или развитие пет проекта.

## 7. Архитектурные принципы и паттерны

В рамках обучения разработчик должен освоить следующие архитектурные принципы:

• SOLID • DRY • KISS • YAGNI

Также необходимо изучить архитектурные подходы:
• MVC
• MVVM
• Flux
• Clean Architecture
• Feature-Sliced Design

## 8. Работа с AI инструментами

AI инструменты должны использоваться как часть ежедневной разработки.

Разработчик должен научиться:
• правильно формировать промпты
• анализировать сгенерированный код
• выявлять ошибки AI
• использовать AI для ускорения разработки

## 9. Soft Skills разработчика

В рамках обучения необходимо развивать следующие soft skills:

• корректная коммуникация в задачах
• описание Merge Request
• участие в code review
• формулирование технических вопросов
• декомпозиция задач
• оценка сроков выполнения

## 10. Контроль и измеримость результатов

Результаты обучения должны быть измеримыми.

Для этого используются:

• еженедельные проверки результатов
• ежемесячные контрольные точки
• ревью архитектурных решений
• анализ качества кода

## 11. Организация обучения

Для организации обучения создаётся отдельный эпик в системе управления задачами.

Внутри эпика создаются задачи:

• изучение материалов
• практические задания
• архитектурные задачи
• контрольные проверки

Все задачи связываются с эпиком обучения сотрудника.
В рамках эпика в комментарии выставляется оценка за материал по пятибальной шкале. Нельзя закрыть эпик при достижении меньше 3х баллов, тема должна быть проработана дополнительно
